# Longform Scale Design

日期：2026-05-13

## 目标

把当前项目从“已验证 20 章长篇链路”推进到“具备支撑千章、百万字以上网络小说长期稳定创作的基础能力”。

这里的“稳定”不是指模型永不犯错，而是指系统能在长周期中持续做到：

- 不丢设定、不污染未来章节、不混淆角色阶段状态。
- 能把正文、摘要、世界事实、伏笔、剧情弧和用户修订组织成可检索上下文。
- AI 生成或分析结果必须可追踪、可审阅、可回滚。
- 千章规模下 UI、检索、审阅和后台任务不会因为数据量增长而失控。

## 当前基线

已验证能力：

- 20 章项目 `霜灯档案：20章Dogfood` 可完整生成正文。
- 当前章节数 20，章节字数求和 78,127。
- Athena 叙事规划包含 20 章规划、4 条故事线、10 条伏笔。
- Athena world-model 有 Profile v1、10 个实体、1 个已确认事实主体。
- 检索索引包含 21 个文档、130 个分块、130 个 embedding，覆盖 20 章正文和 1 条世界事实。
- 世界事实候选包共 20 个，其中 1 个部分批准、19 个待审。
- 自动化验证基线：后端 420 个测试通过，前端 59 个测试文件 / 345 个测试通过。

主要差距：

- 章节、卷、剧情弧之间没有明确的分层记忆模型。
- 默认本地 hash embedding 适合稳定链路验证，不适合作为百万字语义检索上限。
- 每章生成候选事实后，如果全靠人工逐条审阅，千章项目会出现审阅队列爆炸。
- 进程内后台任务不适合千章规模下的批量分析、索引、检查和重试。
- 图谱、时间线、提案列表、章节列表仍需要面向千章数据做分页、折叠、虚拟化和局部视图。
- 项目级统计存在口径不一致风险，例如项目总字数和章节求和应统一。

## 设计原则

1. **分层记忆优先**：千章不能依赖单层上下文，必须有章节、剧情弧、卷、全书四层摘要。
2. **事实必须可审阅**：AI 不直接改世界真相，仍走 proposal/review，但要增加批量去重和低风险合并机制。
3. **检索必须可解释**：上下文包要说明命中了哪些章节、事实、伏笔和剧情弧，不能只给模型一段黑盒文本。
4. **未来章节隔离**：生成第 N 章时，检索和上下文默认不得读取 N 章之后的正文、事实或摘要。
5. **规模化 UI 默认局部化**：千章视图不渲染全量图谱，默认按卷、剧情弧、章节窗口展示。
6. **压测进入主流程**：每个阶段都必须有合成数据压测，不能只靠手工 20 章项目判断。
7. **密钥不入库**：DeepSeek 或其他模型密钥只通过 keyring、环境变量或本地配置使用，不写入仓库、文档、trace 或测试输出。

## 推荐路线

采用“Scale Foundation First”路线。

不先大规模改 UI，也不先增加新的创作花活，而是先建立百万字所需的基础设施：

1. 长篇分层数据模型。
2. 分层摘要生成和重建服务。
3. 查询感知检索上下文包。
4. 世界事实候选聚合、去重、批量审阅。
5. 后台任务持久化和断点续跑。
6. 千章级 UI 局部视图和虚拟化。
7. 百章、三百章、千章合成压测。

## 核心概念

### Work

一个项目内的长篇作品容器。当前 `Project` 已经承担大部分职责，短期不新增独立 `Work` 表，避免过度拆分。

### Volume

卷。网络小说常以卷承载阶段目标和大剧情推进。

字段建议：

- `project_id`
- `volume_index`
- `title`
- `summary`
- `start_chapter_index`
- `end_chapter_index`
- `status`

### Arc

剧情弧。比卷更细，可跨卷，也可嵌在卷内。用于表达副本、阶段反派、修炼阶段、关系推进、悬疑案件等。

字段建议：

- `project_id`
- `arc_key`
- `title`
- `arc_type`
- `start_chapter_index`
- `end_chapter_index`
- `summary`
- `status`
- `stakes`
- `open_threads`
- `resolved_threads`

### Chapter Memory

章节记忆。每章生成后沉淀，不等同于正文。

包含：

- 章节摘要。
- 出场角色。
- 关键事件。
- 世界事实候选。
- 伏笔埋设和回收。
- 状态变化。
- 与前文的依赖。

### Arc Memory

剧情弧记忆。由章节记忆聚合生成，用于中距离上下文。

### Volume Memory

卷级记忆。由剧情弧和章节记忆聚合生成，用于长距离上下文。

### Global Canon

全书规范层。由 Athena 已确认事实、核心设定、长期人物状态、世界规则和不可违背约束组成。

## 上下文分层策略

生成或分析第 N 章时，默认上下文包分为：

1. **全局规范**：核心设定、世界规则、已确认人物身份、不可违背事实。
2. **当前卷摘要**：当前卷目标、冲突、已解决/未解决问题。
3. **相关剧情弧摘要**：与目标章节相关的 arc memory。
4. **最近章节窗口**：默认 N-3 到 N-1 的章节摘要和必要摘录。
5. **查询感知检索**：根据目标章节大纲、出场角色、伏笔和用户指令检索旧章节。
6. **待审风险提示**：相关未审提案、冲突、未闭合伏笔。
7. **用户修订偏好**：最近有效修订反馈、风格偏好、自优化规则。

所有上下文块必须进入 model call trace，并在 UI 可查看。

## 检索设计

短期保留当前 retrieval 表和接口，但增加查询组装层：

- 关键词检索：角色名、地点名、物品名、伏笔关键词、章节号。
- 向量检索：远程 embedding 优先，本地 hash embedding 保底。
- 结构化过滤：只检索 `chapter_index < target_chapter_index`，除非用户明确请求全书复盘。
- 来源混合：正文 chunk、章节记忆、剧情弧记忆、卷记忆、world fact。
- 结果解释：每条命中保留 source type、章节范围、分数、命中原因。

后续可新增远程 embedding 配置，但密钥只走本地安全存储。

## 世界模型审阅设计

千章规模不能让用户逐条审阅所有候选。

需要增加：

- 候选事实规范化：同一角色、同一谓词、同一章节作用域合并。
- 低风险批量合并：例如章节内 presence_count、出场记录、已明确发生事件。
- 高风险单独审阅：死亡、身份反转、规则改写、关系质变、时间线冲突。
- 候选聚类：按角色、地点、剧情弧、章节范围聚合。
- 审阅队列视图：优先显示“会影响后续生成”的高风险项。

原则：低风险降低人工成本，高风险保留人工控制。

## 后台任务设计

短期先在现有 `BackgroundTask` 模型基础上增强，不立即引入 Celery。

第一阶段增加：

- 批量任务进度字段。
- 输入摘要和幂等 key。
- 失败原因和可重试状态。
- 分段执行记录。
- 可恢复的 chapter range 任务。

后续当本地进程内任务不足时，再引入正式队列。

## UI 规模化设计

千章 UI 默认不全量展开。

### 章节列表

- 按卷折叠。
- 支持章节号跳转。
- 支持生成状态、字数、更新时间过滤。
- 超过 200 章启用虚拟列表。

### 叙事图谱

- 默认显示当前卷或当前剧情弧。
- 支持层级：全书总览、卷图、剧情弧图、章节窗口图。
- 伏笔显示生命周期：埋设、推进、回收、逾期。
- 大图只展示聚合节点，禁止默认渲染千章全量节点。

### 提案审阅

- 默认按风险、剧情弧、章节范围聚合。
- 支持批量接受低风险候选。
- 高风险项提供影响范围和相关上下文。

### Athena 对话

- 回答必须显示依据范围。
- 全书级问题默认使用全局摘要、卷摘要、相关剧情弧和检索命中。
- 如果只看到了局部上下文，必须明确说明。

## 压测目标

### P0：20 章回归

必须保持当前 dogfood 能力不退化。

验收：

- 20 章项目可读。
- Athena 主要视图可打开。
- 检索、提案、图谱、时间线无阻塞错误。
- 后端和前端测试通过。

### P1：100 章合成压测

生成或导入 100 章合成正文、摘要、候选事实、检索索引。

验收：

- 章节列表、检索、提案列表响应可接受。
- 构建上下文包时不会读取未来章节。
- 生成第 100 章上下文包包含卷/弧/最近章节/检索证据。

### P2：300 章合成压测

验证中长篇规模。

验收：

- 检索结果可解释。
- 提案聚合后审阅项数量显著小于原始候选数量。
- 图谱默认局部展示，不渲染全量节点。

### P3：1000 章 / 百万字压测

验证目标规模基础。

验收：

- 数据导入、索引、上下文组装可断点续跑。
- UI 不一次性渲染千章节点。
- 查询某角色/伏笔/剧情弧能返回可解释上下文。
- 项目统计和章节求和一致。

## 阶段计划

### Phase 1：规模化基线与分层记忆骨架

目标：

- 新增卷/剧情弧/章节记忆的最小数据结构或服务层。
- 提供 100 章合成数据压测脚本。
- 修正项目统计口径，确保总字数可由章节稳定回填。
- 输出第 N 章上下文包的可测试组装逻辑。

不做：

- 不接入复杂新 UI。
- 不引入正式队列。
- 不替换全部检索实现。

### Phase 2：检索和上下文升级

目标：

- 引入查询感知 retrieval context package。
- 支持远程 embedding 配置路径。
- 增加 source explanation。
- Athena/Hermes 对话都能看到依据范围。

### Phase 3：世界事实候选聚合和审阅降噪

目标：

- 候选事实聚类、去重、风险分级。
- 低风险批量审阅。
- 高风险影响范围视图。

### Phase 4：千章 UI 局部化

目标：

- 章节列表按卷和虚拟化。
- 图谱按卷/弧局部展示。
- 提案列表分页和聚合视图。

### Phase 5：任务队列和断点续跑

目标：

- 增强 BackgroundTask。
- 支持章节范围任务。
- 支持失败重试和分段恢复。

## 第一阶段验收标准

Phase 1 完成时必须满足：

- 后端测试通过。
- 前端测试通过。
- 前端构建通过。
- 新增 100 章合成压测命令。
- 压测可生成章节摘要、卷/弧初步结构和检索索引统计。
- 第 N 章上下文包测试证明不会读取未来章节。
- 项目总字数 API 和章节字数求和一致。

## 风险

- 如果先追求全自动生成千章，质量会不可控。应先建立分层记忆和验证闭环。
- 如果候选事实全部自动入库，世界模型会被噪音污染。必须保留风险分级。
- 如果图谱追求全量可视化，千章下会不可用。必须默认聚合和局部化。
- 如果远程 embedding 失败，应保留本地 hash embedding 作为稳定保底。

## 决策

采用 Phase 1 作为当前实施起点：

1. 建立长篇规模化基线。
2. 新增合成压测能力。
3. 修正统计口径。
4. 建立分层记忆和上下文包骨架。

后续阶段按压测结果推进，不提前实现未验证的复杂功能。
