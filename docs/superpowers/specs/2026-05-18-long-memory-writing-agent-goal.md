# 下一轮长期目标：长记忆网文创作 Agent

本文档用于指导下一次长期 goal。启动 goal 前，应先阅读本文档，并在后续每个阶段以文档为驱动推进。

## 1. 总目标

将 novelv3 从“带 AI 对话能力的写作工具”持续重构为“以对话驱动、专精网络小说创作、具备长期记忆和工具编排能力的写作 Agent 系统”。

目标系统应能支撑百万字、千章级网络小说的长期稳定高质量创作。它不追求成为通用 Agent，而是专注于网络小说领域，围绕长篇创作的连续性、商业节奏、人物弧线、世界设定、伏笔埋收、审稿反馈和持续生成闭环进行优化。

核心方向：

- 对话仍然是用户入口。
- Agent 是系统主脑。
- Hermes、Athena、世界模型、检索、知识库、审稿、任务队列、Trace 等模块都应成为 Agent 可调用的工具或能力。
- 系统通过真实生成长篇小说来发现问题，而不是空泛优化。

## 2. 下一轮 goal 的主线任务

下一轮 goal 的主线不是“抽象优化项目”，而是持续真实生成一部长篇网络小说，并以此驱动系统演进。

理想目标：

- 持续生成一部结构合理、可读性稳定的长篇小说。
- 单章正文不少于 2000 字。
- 长期目标不少于 600 章，并为千章级扩展保留能力。
- 生成过程必须持续暴露并修复系统问题。

工作闭环：

```text
真实生成小说
  -> 审查章节质量与系统体验
  -> 记录问题和弱点
  -> 修复或优化模块
  -> 做合适粒度的验证
  -> 继续生成小说
```

这部长篇小说既是产物，也是压测样本、质量基准和系统演进驱动器。

## 3. Agent 架构方向

下一轮不要求一次性设计出完美架构，但所有优化应朝向以下形态收敛：

```text
用户对话
  -> Writing Agent Core
      -> 意图理解
      -> 长期记忆召回
      -> 任务拆解
      -> 工具选择
      -> 结果审查
      -> 自优化沉淀
  -> 工具层
      -> Hermes 创作执行
      -> Athena / 世界模型
      -> Retrieval 检索记忆
      -> Knowledge Base 知识库
      -> Review 审稿
      -> Task Queue 长流程任务
      -> Trace / Context 审计
      -> Export / Manuscript 输出
```

关键原则：

- Agent 不应吞掉所有模块细节。
- Agent 负责编排、计划、调用、判断和沉淀。
- 各模块保持专业职责，通过清晰契约成为 Agent 工具。
- 不要把 Agent 做成一个超级大函数。

## 4. 模块边界

### 4.1 Hermes

Hermes 应定位为创作执行工具，负责构思、章节生成、续写、改写、批量生产和风格执行。

### 4.2 Athena / 世界模型

Athena 应定位为小说内部事实与一致性工具。

它负责：

- 世界真相。
- 角色、地点、势力、规则、事件。
- 角色认知。
- 章节快照。
- 提案审批。
- 冲突检查。

世界模型回答的问题是：这部小说内部什么是真的。

### 4.3 Retrieval

检索层应定位为长篇上下文召回工具。

它负责从正文、世界事实、历史决策、知识库和写法模式中召回当前任务所需上下文。

### 4.4 Knowledge Base

知识库不是世界模型的替代品。它是 Agent 的长期创作记忆系统。

它回答的问题是：这本书应该怎么写，用户长期想要什么，系统从过往创作和参考作品中学到了什么。

知识库至少包含：

- 作者记忆：用户偏好、风格要求、禁忌、反复纠正的问题。
- 项目知识：本书定位、目标读者、商业策略、章节要求、风格基准。
- 拆书库：参考作品的结构拆解、节奏模式、人物关系推进、伏笔设计方法。
- 写法模式库：从拆书和创作经验中抽象出的可复用技法。
- 自优化经验：生成失败、审稿问题、用户反馈沉淀出的长期规则。

知识库可以影响章节生成、审稿和世界模型提案，但不应直接污染世界真相层。

### 4.5 Review

审稿模块应成为 Agent 的质量判断工具，而不是可有可无的辅助页面。

审稿至少覆盖：

- 剧情连续性。
- 人物一致性。
- 世界设定一致性。
- 伏笔埋收。
- 商业节奏。
- 章节可读性。
- 是否像大纲而非正文。
- 章节字数和场景密度。

### 4.6 Task Queue

任务队列应支撑长流程自动化：

- 批量生成章节。
- 批量审稿。
- 批量修订。
- 世界模型回灌。
- 检索索引更新。
- 知识库沉淀。
- 阶段报告生成。

长篇创作不能依赖一次性同步请求。

## 5. 参考项目拆解

本项目已在本地保存三个外部 Agent 项目的源码快照，作为后续学习材料：

- `references/agent-projects/openclaw/`
- `references/agent-projects/hermes-agent/`
- `references/agent-projects/openhuman/`

这些项目只作为参考，不作为 novelv3 的依赖或直接复制对象。

下一轮 goal 中应重点拆解它们的：

- 长期记忆结构。
- Agent 工具调用与编排。
- 计划、执行、反思循环。
- 上下文压缩与召回策略。
- 用户偏好、项目记忆、程序性记忆边界。
- 多工具、多任务、多代理协作方式。

注意：

- 不要照搬通用 Agent 架构。
- novelv3 只需要专精网络小说写作。
- 拆解结果应转化为 novelv3 自己的设计文档、接口契约和阶段计划。

## 6. 拆书能力

拆书能力是知识库和自优化系统的重要输入。

目标不是复制热门小说，也不是让模型仿写原文，而是分析并抽象优秀作品的结构规律。

拆书应关注：

- 开篇钩子。
- 卷结构。
- 章节节奏。
- 冲突推进。
- 爽点设计。
- 悬念和反转。
- 人物关系递进。
- 伏笔埋收。
- 情绪曲线。
- 商业类型套路。

拆书结果应进入：

- 项目知识。
- 写法模式库。
- 审稿标准。
- 章节生成约束。
- 自优化经验。

版权边界：

- 不应长期保存大段受版权保护的原文作为生成材料。
- 应优先保存结构化分析、抽象模式和系统自己的总结。

## 7. 文档驱动流程

下一轮 goal 必须采用文档驱动执行。

### 7.1 启动前

启动 goal 后第一件事：

1. 阅读本文档。
2. 检查当前 `main` 状态。
3. 创建本轮 goal 的执行目录。
4. 写出阶段 1 计划文档。
5. 再开始具体实现或狗粮测试。

建议目录：

- `docs/superpowers/specs/`：当前仍有效的总体设计。
- `docs/superpowers/plans/long-memory-agent/`：下一轮 goal 的阶段计划。
- `docs/superpowers/notes/long-memory-agent/`：阶段报告、狗粮记录、审稿记录、问题清单。

如果目录不存在，应先创建。

### 7.2 每个阶段必须有阶段文档

每个阶段开始前，必须写或更新阶段文档，内容包括：

- 阶段编号。
- 阶段目标。
- 本阶段要生成或验证的小说内容。
- 本阶段要改进的系统能力。
- 涉及模块。
- 预期风险。
- 验证层级。
- 不做什么。

每个阶段结束后，必须在阶段文档中补充：

- 实际完成内容。
- 生成小说进度。
- 发现的问题。
- 已修复的问题。
- 未修复但记录的问题。
- 运行过的验证命令。
- 下一阶段建议。

上下文压缩后，应优先从这些文档恢复工作状态。

## 8. 验证节奏

上一轮 goal 的主要效率问题是：每个 phase 做很少改动就跑完整前后端测试。下一轮应保留谨慎，但改用分层验证。

### T0：文档或无风险改动

适用：

- 只改文档。
- 只整理计划。
- 不影响运行时代码。

验证：

- `git diff --check`
- 必要时检查链接和路径。

### T1：局部功能改动

适用：

- 单个服务。
- 单个 API。
- 单个前端组件。
- 单个提示词或质量门禁。

验证：

- 相关单测。
- 相关 API smoke。
- 必要时浏览器局部验证。

### T2：模块级改动

适用：

- 影响一个完整模块。
- 涉及前后端契约。
- 涉及章节生成、检索、世界模型、队列等关键路径。

验证：

- 相关后端测试集。
- 相关前端 vitest。
- TypeScript 或 build。
- 关键浏览器流程。
- 必要的长篇 smoke。

### T3：里程碑或高风险改动

适用：

- 合并前。
- 大架构改动。
- 数据模型迁移。
- Agent Core、任务队列、世界模型、检索主路径变化。
- 阶段性稳定版本。

验证：

- 后端全量测试。
- 前端全量测试。
- TypeScript / build。
- E2E。
- 长篇规模 smoke。
- 真实章节生成回归。

原则：

- 不为了形式每个小阶段都跑完整验证。
- 不在高风险改动后省略完整验证。
- 阶段文档必须记录为什么选择当前验证层级。

## 9. 子代理与团队协作

下一轮 goal 应更主动使用子代理和团队化分工。

建议角色：

- 主控 Agent：维护目标、计划、取舍和最终集成。
- 小说主笔：负责章节生成质量。
- 连续性编辑：检查剧情、设定、人物一致性。
- 世界模型管理员：审查事实、提案、角色认知和时间线。
- 拆书分析员：拆解参考作品和写法模式。
- 读者视角审稿：判断章节吸引力、节奏和阅读体验。
- 架构审查员：评估 Agent 化重构是否过度或偏离目标。
- 验证代理：运行局部或全量验证，检查回归风险。

子代理不只用于编码，也应用于质量审查、拆书分析、阶段总结和验证分工。

## 10. 质量指标

下一轮 goal 应持续跟踪以下指标：

- 已生成章节数。
- 每章字数是否达到 2000 字以上。
- 章节是否为正文而非大纲。
- 章节连续性问题数量。
- 人物漂移问题数量。
- 世界设定冲突数量。
- 伏笔未跟踪数量。
- 检索召回是否命中关键前文。
- 世界模型是否及时回灌。
- 审稿问题是否能转为可执行修复。
- 用户偏好是否进入长期记忆。
- 前端在大量章节下是否仍可用。
- 任务队列是否可恢复、可观察、可追踪。

这些指标不要求一开始全部自动化，但阶段文档中应持续记录。

## 11. 约束

- 不要在文档中写入 API key 或其他密钥。
- 需要调用大模型时，使用本地环境变量或用户明确提供的运行时配置。
- 不要把第三方参考源码直接纳入 Git 历史。
- 不要一次性做大而空的重构。
- 不要让知识库绕过提案机制直接修改世界真相。
- 不要用拆书结果生成侵权式仿写。
- 不要为了追求 Agent 化而破坏现有可运行能力。
- 每个阶段都应能说明：本阶段为什么服务于长篇稳定创作。

## 13. 成功标准

下一轮 goal 不是以某个单一功能上线作为成功标准，而是以系统逐步具备以下能力作为成功标准：

- AI 从普通对话助手转向长期记忆写作 Agent。
- 对话能驱动真实长篇创作，而不是只回答问题。
- Agent 能调用并编排现有模块。
- 用户偏好、项目策略、拆书模式和创作经验能长期沉淀。
- 世界模型继续保证小说内部事实一致性。
- 真实生成的小说章节质量可被审查、修订和持续提升。
- 项目验证节奏更合理，不牺牲质量，也不浪费大量时间。
