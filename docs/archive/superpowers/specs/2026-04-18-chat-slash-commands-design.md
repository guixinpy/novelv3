# 墨舟聊天命令与上下文管理设计

- 日期：2026-04-18
- 主题：聊天区 `/命令` 体系、`/clear`、`/compact`
- 范围：项目详情页聊天区、聊天协议、对话消息存储、生成命令触发链路
- 结论：移除底部快捷按钮，改为 Claude Code 风格 slash command；新增真实持久化的 `/clear` 与 `/compact`；未注册命令一律回落为普通文本发送给 AI

## 1. 目标与非目标

目标：

- 将聊天区从“按钮驱动 + 普通输入”升级成“单输入框 + slash command”模式。
- 提供像 Claude Code 一样的上下文管理能力：`/clear` 清空上下文，`/compact` 压缩上下文。
- 让命令体系成为真实前后端协议，而不是前端假动作，保证刷新后状态一致。
- 把现有 `生成设定 / 生成故事线 / 生成大纲` 三个按钮并入命令体系，支持附加自由提示词。

非目标：

- 第一轮不做 `/help`、命令别名、多级子命令。
- 不做任意 slash 命令自动发现或模型自解释命令。
- 不扩散到项目列表页、设置页或其他非聊天入口。

## 2. 方案选择

排除方案：

- 前端本地假命令：便宜，但刷新后状态丢失，和后端真实对话历史脱节。
- 全交给模型理解 `/命令`：最不稳，控制指令不该靠模型猜。

采用方案：

- 前端只负责命令发现、补全和已注册命令识别。
- 后端负责命令分发、上下文清理、上下文压缩、命令消息持久化。
- 对话历史继续是唯一事实来源，`/compact` 后的摘要也作为一条真实消息存储。

## 3. 命令行为定义

第一期命令集：

- `/clear`
- `/compact`
- `/setup`
- `/storyline`
- `/outline`

识别规则：

- 输入以 `/` 开头时，前端只会拦截“已注册命令”。
- 例如 `/setup 主角是植物学家` 会识别成命令。
- `/foo bar`、`/今天想聊聊` 这类未注册写法，不提示错误，直接当普通文本发给 AI。
- 前端不解析命令参数语义，只透传原始参数字符串。

### 3.1 `/clear`

作用：

- 清空当前项目的对话历史和待确认动作，但不删除任何项目业务数据。

执行结果：

- 删除当前项目全部 `dialog_messages`
- 清空 `dialog.pending_action_id`
- 将 `dialog.state` 重置到 `chatting`
- 写入一条新的系统消息：`上下文已清空，你可以开始新的对话。`

约束：

- `RUNNING` 状态下禁止执行，避免后台生成结果和对话状态脱钩。
- `PENDING_ACTION` 状态下允许执行，用作“一键放弃当前待确认动作并清空历史”。

### 3.2 `/compact`

作用：

- 将当前对话压缩成一条摘要消息，替代被压缩的旧上下文。

执行结果：

- 选择“上一条摘要之后”的所有普通消息作为本轮压缩范围。
- 生成一条 `summary` 类型消息，标题为 `上下文摘要`。
- 删除本轮被压缩的旧消息。
- 保留该摘要消息以及之后的新消息。

交互结果：

- 聊天流中插入一张默认折叠的“上下文摘要卡”。
- 卡片可展开查看摘要正文。
- 后续发给模型的上下文只携带摘要，而不再携带被删除的旧消息。

最小压缩条件：

- 如果上一条摘要之后没有足够消息可压缩，则不生成新摘要。
- 返回一条系统消息：`当前没有可压缩的新上下文。`

失败策略：

- 优先尝试模型摘要。
- 如果模型不可用或调用失败，回落到 deterministic fallback 摘要。
- fallback 至少覆盖：
  - 最近用户目标
  - 最近已执行或待执行动作
  - 当前项目诊断
  - 最近一轮补充要求

约束：

- `RUNNING` 和 `PENDING_ACTION` 状态下都禁止执行，避免压缩掉正在使用的交互上下文。

### 3.3 `/setup` `/storyline` `/outline`

作用：

- 完全替代当前聊天区底部三个按钮。

形式示例：

- `/setup`
- `/setup 主角是植物学家，世界观偏硬科幻`
- `/storyline 主线冲突要更集中`
- `/outline 12章，节奏快一些`

行为：

- 仍然进入当前已有的待确认流。
- 后端创建 `pending_action`，但 `params` 内额外写入 `command_args`。
- 待确认文案必须包含附加要求，例如：
  - `准备生成设定，并附加要求：主角是植物学家。确认执行吗？`

确认后：

- 原有生成链路继续负责生成。
- 对应 prompt 会接收 `command_args`，让命令比旧按钮更灵活，而不是仅仅换入口。

## 4. 数据结构与协议

### 4.1 前端命令注册表

新增 `chatCommands` 注册表，作为唯一命令定义源。

每个命令至少包含：

- `name`
- `label`
- `description`
- `example`
- `supportsArgs`

用途：

- 输入框 slash 菜单
- 提交解析
- 快速补全

### 4.2 聊天请求协议扩展

现有 `ChatIn` 增加：

- `input_type: 'text' | 'button' | 'command'`
- `command_name?: string`
- `command_args?: string`

原则：

- 所有命令仍走 `/api/v1/dialog/chat`
- 不新增平行命令 API

### 4.3 消息结构扩展

现有消息模型增加：

- `message_type: 'plain' | 'summary' | 'command'`
- `meta: JSON | null`

约束：

- 普通聊天默认 `plain`
- `/compact` 生成的摘要消息使用 `summary`
- `/clear` 和命令回执可使用 `command` 或继续使用 `system + plain`；第一轮允许保持简单，但摘要必须有明确 `summary`

`summary.meta` 最小字段：

- `title`
- `summary_text`
- `compacted_count`
- `command_name`

### 4.4 数据库存储

采用最小改法：

- 给 `dialog_messages` 增加 `message_type`
- 给 `dialog_messages` 增加 `meta`

不新增摘要表。

原因：

- 摘要本质就是一条消息
- 历史加载、排序、删除可以继续复用现有消息流
- 最小侵入，维护成本最低

## 5. UI 设计

### 5.1 输入区

聊天区底部只保留一个输入框，不拆分命令输入和普通输入。

占位文案调整为：

- `输入消息，或键入 / 查看命令`

发送逻辑：

- 命中已注册命令 -> 走命令协议
- 其他输入 -> 继续作为普通文本发给 AI

### 5.2 Slash 菜单

触发条件：

- 输入首 token 以 `/` 开头，例如 `/`、`/co`、`/setup 主角...`

不触发条件：

- 输入中间出现 `/`
- 非首 token 的 `/`

交互：

- 菜单浮在输入框上方
- 默认列出全部命令
- 随前缀过滤，如 `/co` 仅显示 `/compact`
- 支持 `↑ / ↓` 选择、`Enter` 补全、`Esc` 关闭、鼠标点击

每一项展示：

- 命令名，例如 `/compact`
- 一句话作用，例如 `压缩当前对话为摘要`
- 示例，例如 `/outline 12章，节奏快`

补全规则：

- 仅补全命令头与一个空格
- 不自动填参数模板

### 5.3 摘要卡

`/compact` 后生成独立摘要卡，而不是普通消息气泡。

卡片内容：

- 标题：`上下文摘要`
- 副信息：`已压缩 N 条消息`
- 提示：`点击展开查看摘要`

交互：

- 默认折叠
- 点击整卡或箭头展开
- 展开后显示摘要正文
- 可再次折叠

状态约束：

- 展开/折叠状态仅存在前端本地 UI
- 不做后端持久化

### 5.4 命令消息显示

命令消息保留在对话流中。

示例：

- 用户输入 `/outline 12章，节奏快一些`
- 聊天流里先出现用户命令消息
- 再出现系统或助手回执

原因：

- 保持会话可回溯
- 便于用户回看“当时具体下了什么命令”

### 5.5 旧按钮下线

完全移除：

- `生成设定`
- `生成故事线`
- `生成大纲`

不保留并存过渡期。

原因：

- 双入口会制造双心智
- 命令式入口已经覆盖其职责，且支持参数，体验更强

## 6. 状态与边界条件

### 6.1 命令冲突规则

- `RUNNING` 时：
  - 禁止 `/clear`
  - 禁止 `/compact`
  - 禁止新的生成命令
- `PENDING_ACTION` 时：
  - 禁止 `/compact`
  - 禁止新的生成命令
  - 允许 `/clear`

### 6.2 `/clear` 作用边界

`/clear` 只清理当前项目会话，不触碰：

- 项目本体
- 设定 / 故事线 / 大纲 / 正文
- 版本记录
- 已生成业务结果

### 6.3 非法命令处理

不做“未知命令”报错 UI。

规则：

- 未注册 slash 输入一律回落为普通文本发送给 AI

理由：

- 更符合 Claude Code / Codex 直觉
- 避免把 slash 当成特殊危险前缀

## 7. 前后端实现要点

### 7.1 后端

- 扩展 `ChatIn`、`ChatMessageOut`
- 扩展 `DialogMessage` 模型与迁移
- 在 `chat` 接口内新增命令分发
- 实现 `/clear`
- 实现 `/compact`
- 在待确认描述中纳入 `command_args`
- 在生成 prompt 中接入 `command_args`

### 7.2 前端

- 新增命令注册表
- 在聊天 store 或输入解析层增加 slash 命令判断
- 移除 `QuickActions`
- 在 `ChatWorkspace` 增加 slash 菜单
- 在 `ChatMessage` 增加 `summary` 卡片渲染
- 保证刷新后摘要卡仍按消息类型正确还原

## 8. 测试要求

后端必须覆盖：

- 已注册命令识别成功
- 未注册 slash 输入回落为普通文本
- `/clear` 删除消息、清空 pending action、重置 dialog state
- `/compact` 生成摘要消息并删除被压缩消息
- 多次 `/compact` 仅压缩上一条摘要之后的增量消息
- `/setup xxx`、`/storyline xxx`、`/outline xxx` 会把参数带入 `pending_action.params`
- 确认执行后生成 prompt 收到 `command_args`

前端必须覆盖：

- slash 菜单显示、过滤、键盘导航、补全、关闭
- 已注册命令提交时走命令分支
- 未注册 slash 输入走普通文本分支
- 摘要卡默认折叠、可展开查看
- 旧按钮已被移除

浏览器验收必须覆盖：

- 输入 `/` 后看到命令菜单
- 旧按钮不存在
- 执行 `/compact` 后出现摘要卡
- 刷新页面后摘要卡仍存在
- `/foo` 会作为普通文本进入 AI 对话

## 9. 实现顺序

1. 后端 schema、模型、迁移补齐
2. 后端命令分发与 `/clear` `/compact`
3. 生成链路接入 `command_args`
4. 前端命令注册表与输入解析
5. 前端 slash 菜单、摘要卡、旧按钮移除
6. 单元测试、浏览器 smoke、全量回归

## 10. 刻意不做的内容

第一期不做：

- `/help`
- 命令别名
- 多级子命令
- 命令权限系统
- 摘要展开状态后端持久化

原因：

- 先把最核心的命令心智和上下文管理打实
- 避免命令体系第一轮就长成难以维护的垃圾场
