# 设定 Inspector 结构化改造设计

- 日期：2026-04-18
- 主题：右侧 Inspector `设定` 面板重构
- 范围：`frontend/src/components/tabs/SetupTab.vue` 及其配套前端类型、展示组件、测试
- 结论：保留右侧 Inspector 顶层 `设定` 入口，在 `设定` 面板内部新增二级 tabs：`角色 / 世界观 / 核心概念`；前端基于已有后端结构化字段渲染信息卡，不再展示黑盒字符串或 JSON dump

## 1. 目标与非目标

目标：

- 让 `设定` 面板从“黑盒字符串堆”变成可读、可定位、可针对性发起更新的结构化视图。
- 保留右侧 Inspector 顶层导航稳定性，不继续膨胀顶层 tabs。
- 充分利用现有后端 `setup` 数据结构，不为展示问题先引入额外接口改造。
- 顺手收紧前端 `setup` 类型边界，减少 `any` 扩散。

非目标：

- 本轮不在右侧直接提供可编辑表单。
- 本轮不新增后端字段，也不改 AI 生成协议。
- 本轮不把 `角色 / 世界观 / 核心概念` 提升为 Inspector 顶层 tabs。

## 2. 当前问题

当前 `SetupTab` 存在三个核心问题：

- 角色信息几乎不可用。真实数据里已经有 `age / gender / personality / background / goals / character_status`，前端却只显示名字和一小段描述，用户无法判断该对哪个角色下什么更新指令。
- 世界观和核心概念直接 `JSON.stringify`，等于把结构化数据重新打碎成原始字符串，阅读成本高，视觉也糟。
- 组件边界松散，前端把 `setup` 当 `any` 传来传去，不利于后续持续优化。

结论：

- 问题根子在展示层，不在接口能力。
- 第一轮应该优先改前端信息架构，而不是盲目扩后端。

## 3. 核心决策

采用“顶层稳定 + 面板内分层”的结构：

- 右侧 Inspector 顶层仍然只有一个 `设定` tab。
- `设定` 面板内部新增二级 tabs：`角色 / 世界观 / 核心概念`。
- 二级 tabs 使用紧凑 segmented pills，不使用左侧树或长页折叠组。

不采用以下方案：

- 不把三块内容拆成 Inspector 顶层 tabs。
  原因：会继续挤压右侧主导航，破坏刚做完的精简头部。
- 不做单页长滚动。
  原因：右侧面板天生窄，长滚动会让定位效率很差。

## 4. 信息架构

### 4.1 二级 tabs

`设定` 面板内部固定三项：

- `角色`
- `世界观`
- `核心概念`

选择规则：

- 首次进入 `设定` 面板默认选中 `角色`。
- 同一项目内切换二级 tab 时保持本地选择。
- 切换项目后重置回 `角色`，避免跨项目残留子视图状态。

### 4.2 角色视图

角色是最需要持续补充和更新的部分，因此采用“列表 + 详情”的结构，而不是单纯卡片堆。

桌面端：

- 左侧为角色列表。
- 右侧为当前选中角色的详情卡。

窄屏或容器较窄时：

- 列表在上，详情在下，垂直堆叠。

角色列表项最少展示：

- 姓名
- 一行摘要：优先 `background`，否则 `personality`，否则 `goals`，都没有则显示“待补充角色摘要”
- 状态标签：如 `alive`
- 辅助标签：年龄、性别；缺失则不强行显示无意义占位

角色详情卡展示：

- 姓名
- 年龄 / 性别 / 状态
- 性格
- 背景
- 目标

字段策略：

- 字段为空时明确显示“待补充”，不留纯空白。
- 即使角色只有名字，也要能渲染成完整详情框架，让用户一眼看出还缺什么。

### 4.3 世界观视图

世界观基于现有后端字段拆成 5 张信息卡：

- 时代背景：`background`
- 地理格局：`geography`
- 社会结构：`society`
- 规则体系：`rules`
- 氛围基调：`atmosphere`

展示原则：

- 每块为独立卡片，不再输出原始 JSON。
- 卡片标题直译为用户能理解的文案，不暴露字段名。
- 空字段显示“待补充”，并保留卡片位置，避免用户误以为该维度不存在。

### 4.4 核心概念视图

核心概念拆成 4 张信息卡：

- 主题：`theme`
- 前提设定：`premise`
- 核心钩子：`hook`
- 独特卖点：`unique_selling_point`

展示原则：

- 比世界观更强调“创作抓手”，文案样式更聚焦阅读和定位。
- 依然不直接暴露 JSON 或英文字段名。

## 5. 视觉与交互要求

视觉方向继续服从现有羊皮纸体系，但要避免把 `设定` 面板做成另一套后台界面。

要求：

- 二级 tabs 视觉上低于 Inspector 顶层 tabs，不与主导航抢层级。
- 内容卡使用干净浅底，保证长文本可读。
- 角色列表项必须可一眼区分选中态，不能只靠微弱边框。
- 长文本默认完整展示，允许自然换行，不用折叠成“查看更多”。

交互要求：

- 角色列表点击后更新详情区。
- 如果当前选中角色在新数据刷新后不存在，则回退到第一名角色。
- 没有角色时，仍展示空状态卡，而不是整个 tab 留白。

## 6. 类型与组件边界

这轮要顺手收紧前端类型，不继续传播 `any`。

建议新增或补齐前端类型：

- `SetupData`
- `SetupCharacter`
- `SetupWorldBuilding`
- `SetupCoreConcept`

组件边界：

- `SetupTab.vue`
  负责二级 tabs 壳层、子视图选择、响应式布局拼装
- `SetupSectionTabs.vue`
  负责 `角色 / 世界观 / 核心概念` 的二级 tab 切换
- `SetupCharactersPanel.vue`
  负责角色列表 + 详情
- `SetupWorldPanel.vue`
  负责世界观字段卡片
- `SetupConceptPanel.vue`
  负责核心概念字段卡片
- `setupPresentation.ts`
  负责字段标签、空值归一化、角色摘要提取等纯展示逻辑

约束：

- 不把所有逻辑继续堆回一个 `SetupTab.vue` 大文件。
- 不在模板里塞大段字段兜底逻辑。

## 7. 数据策略

第一轮完全复用现有后端返回：

- `characters: CharacterProfile[]`
- `world_building: WorldBuilding`
- `core_concept: CoreConcept`

前端负责做展示归一化：

- 缺字段 -> “待补充”
- 空字符串 -> “待补充”
- 异常值或未知类型 -> 安全转字符串，但不直接 `JSON.stringify` 整块对象

如果未来 AI 输出质量波动导致结构污染，再单独补后端 schema 校验；这不属于本轮必要改动。

## 8. 测试策略

最少需要补这些验证：

- 二级 tabs 切换正常，默认停在 `角色`
- 角色列表点击后详情区正确切换
- 空字段会展示“待补充”
- 世界观和核心概念不再输出 JSON 字符串
- 构建通过，浏览器中 `设定` 面板能正常切换和渲染

建议测试形态：

- Vitest + Vue Test Utils：覆盖 `SetupTab` 和展示归一化逻辑
- `npm run build`
- `agent-browser` 浏览器回归：真实查看 `设定` 面板层级和内容表现

## 9. 风险与取舍

风险：

- 某些项目的 `setup` 数据可能不完整，角色只有名字，其他字段全空。
- 长文本较多时，右侧面板容易再次被做得又重又挤。

处理：

- 用“待补充”占位把缺口显式暴露给用户，而不是藏起来。
- 角色采用“列表 + 单详情”而不是“所有角色全展开”，控制面板高度。
- 世界观和核心概念采用字段卡而不是大段原文混排，压住阅读复杂度。

最终取舍：

- 优先可读性、可定位性和后续可更新性。
- 暂不追求直接编辑，避免把 Inspector 变成半成品表单系统。
