# Athena UI/UX Information Architecture Design

## Status

Approved in discussion on 2026-04-30.

This document captures the agreed Athena frontend direction before implementation. It is a design baseline, not an implementation plan.

## Goal

把 Athena 从“后端表和 API 的直接导航”改成创作者可理解的世界模型工作台。

Current pain points:

- 左侧导航把角色、地点、势力、物品、关系、规则等叶子数据直接摊开，入口冗杂。
- 角色、地点、势力等节点目前只显示名称、类型、描述，丢失后端已有的丰富设定信息。
- 真相、角色认知、提案、检索、检查等概念混在同一层级，用户难以判断哪些是 canon、哪些是候选、哪些只是工具输出。

## Design Principles

- 完整信息必须可达：后端已保存的节点核心信息，前端必须有明确展示位置。
- 展现形式要分层：完整不等于裸露数据库字段，默认视图按创作者理解路径组织。
- 待审提案不能和已确认真相混淆：候选变更必须始终带有审核状态。
- 关系是边，规则是约束：它们可以出现在设定库中，但不能伪装成普通节点。
- 工具不占主导航：检索、章节分析、自优化、Athena 对话是辅助能力，不是主数据类。

## Top-Level Athena Sections

Athena 内部一级入口收敛为 5 类：

```text
总览
设定库
叙事脉络
真相认知
待审变更
```

### 总览

总览不是数据类，只回答三件事：

- 世界模型是否健康。
- 当前有什么待处理。
- 下一步应该做什么。

可保留导入 Setup、分析最新章节、处理待审提案等行动入口，但不应把总览变成多个工具的堆叠页。

### 设定库

设定库承载世界对象、对象之间的边，以及世界运行约束。

内部视图：

```text
节点
图谱
规则
```

节点筛选：

```text
全部 / 角色 / 地点 / 势力 / 物品 / 资源 / 概念
```

关系不再作为左侧一级入口。关系通过节点详情、图谱视图和关系筛选呈现。

规则不并入节点。规则是影响节点和事件的约束，按范围、来源、是否生效、相关节点和证据展示。

### 叙事脉络

叙事脉络把时间、剧情和章节计划放在同一大区，但内部语义必须区分。

内部视图：

```text
时间线
故事线
章节结构
伏笔线
```

边界：

- 时间线：世界内事件顺序，回答“实际发生了什么”。
- 故事线：剧情、主题、冲突和线索推进，回答“故事怎么推进”。
- 章节结构：大纲、章节目标、章节功能，回答“每章写什么、承担什么作用”。
- 伏笔线：伏笔埋设、误导、回收和信息释放策略。

### 真相认知

真相认知是 Athena 的核心区，负责区分事实、canon、角色视角和信息披露。

内部视图：

```text
事实账本
真相投影
主体认知
信息披露
```

边界：

- 事实账本：原始事实声明，包含层级、状态、证据、章节、生效时间和失效时间。
- 真相投影：当前 canon 世界状态，只读展示，不混入待审提案。
- 主体认知：某个角色或主体知道什么、误解什么、能看到哪些事实。
- 信息披露：读者和角色何时知道某个真相，哪些伏笔已埋未回收。

### 待审变更

待审变更承载世界模型治理流程。

内部视图：

```text
提案队列
影响范围
冲突检查
审核记录
```

边界：

- 提案是候选，不是 canon。
- 通过后才进入事实账本或正式节点数据。
- 冲突检查服务审核，不等于自动修改。
- 审核、回滚和编辑记录必须可追溯。

## Set Library Layout

设定库采用三栏工作台：

```text
左：节点筛选列表
中：完整节点详情
右：关系 / 时间事件 / 真相认知 / 待审上下文
```

Rationale:

- 比卡片库更适合深度阅读节点。
- 比图谱优先更稳定，不会在复杂小说中变成难读的关系线团。
- 支持浏览、比较、追证据、处理待审变更的连续工作流。

## Node Detail Model

节点必须展示完整信息，但按层级组织。

通用骨架：

```text
基础身份
创作摘要
关系网络
事实账本
叙事关联
待审变更
```

展示层级：

1. 摘要层：名称、类型、定位、完整度、待审状态、冲突状态。
2. 创作理解层：按节点类型展示最有助于创作者理解的信息。
3. 结构关系层：连接节点、关系类型、强度、公开/隐藏、生效时间。
4. 事实账本层：已确认事实、事实来源、证据、章节、生效/失效时间。
5. 叙事关联层：相关时间线、故事线、章节、大纲、伏笔。
6. 待审变更层：相关提案、冲突、一致性问题。

### Node Type Modules

角色：

- 身份、别名、公开形象。
- 动机、恐惧、底线。
- 能力、限制、隐藏真相。
- 所属势力、亲密/敌对关系。
- 出场章节、角色弧光。

地点：

- 空间层级、功能。
- 进入条件、危险、资源。
- 所属势力、常驻角色。
- 发生事件、叙事用途。

势力：

- 目标、doctrine、组织结构。
- 成员规则、权力结构。
- 资源、地盘、敌友关系。
- 公开形象、隐藏议程。

物品：

- 来源、用途、激活条件。
- 使用代价、限制、副作用。
- 持有者、流转记录。
- 相关事件、伏笔作用。

资源/概念：

- 定义、作用范围。
- 约束条件、例外。
- 影响的节点、事件、规则。

## Right Context Rail

设定库右栏固定为 4 块：

```text
关系摘要
时间与叙事
真相/认知状态
待审变更
```

关系摘要：

- 连接节点。
- 关系类型。
- 公开/隐藏。
- 强度或状态。

时间与叙事：

- 出场章节。
- 相关事件。
- 相关故事线。
- 相关伏笔。

真相/认知状态：

- 公开事实。
- 隐藏事实。
- 哪些角色知道。
- 是否存在误解或信息差。

待审变更：

- 相关提案。
- 冲突。
- 一致性问题。
- 是否阻断 canon 更新。

## Tools And Secondary Entrances

以下能力不进入主导航：

- 检索：做成全局搜索或证据抽屉，服务节点、事实、提案和章节。
- 章节分析：作为上下文按钮，执行后进入待审变更。
- 一致性检查：进入待审变更的冲突检查视图，也可作为节点/章节/提案的上下文动作。
- 自优化：归入写作偏好或系统学习洞察，不作为世界模型主数据。
- Athena 对话：保留为交互入口，可生成提案或解释当前上下文，但不是数据分类。

## Compatibility And Migration Notes

- 旧路由如 `/athena/characters`、`/athena/locations` 应兼容跳转到设定库节点筛选，而不是直接失效。
- `AthenaSection` 不应继续固化为所有叶子页枚举。后续实现应改为一级 section + 二级 view/filter。
- 合并角色、地点、势力、物品时，加载层不应因筛选变化重复拉取 ontology。
- Setup 草稿、正式 profile、真相投影和待审提案必须明确来源和状态。
- UI 命名要面向创作者，避免把 `retrieval`、`projection`、`optimization` 等系统词直接暴露成主入口。

## Testing Expectations

设计进入实现后至少覆盖：

- Athena 导航渲染和旧路由兼容。
- 设定库节点筛选不重复加载 ontology。
- 节点详情能展示不同节点类型的专属模块。
- 真相投影不混入待审提案。
- 待审变更入口显示提案状态、冲突和审核记录。
- 前端构建通过。

## Non-Goals

- 本设计不要求立即新增后端数据模型。
- 本设计不要求一次性实现图谱可视化的复杂交互。
- 本设计不改变 world-model 的 canon 写入规则。
- 本设计不把 Athena 对话升级成第二套编辑入口。
